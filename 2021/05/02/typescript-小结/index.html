TypeScript—小结

xucongcong
2021-05-02
  • TypeScript
  • JavaScript

TypeScript可以看成JavaScript的一个超集。

优点：

  • 功能更强大
  • 生态更健全
  • 渐进式开发（即使不知道新语法，旧语法依然可用）

缺点：

  • 多了很多概念，如接口、泛型、枚举
  • 初期项目，typescript会增加一些成本（需要编写很多对象声明）

# 1. 安装和配置

npm install -g typescript
npm install -g ts-node
npm install -g tslib @types/node

在typescript中导入npm模块，而这些模块可能并不是typescript编写的，没有类型声明文件，所以其提供的成员没有强类型体验。可以使用单独的declare语句声明，用于兼容普通js模块。不过typescript社区非常强大，大部分模块的声明文件都可以下载，一般要安装文件会有提示：

npm install @types/lodash -D
  • 安装到开发环境：npm install typescript —dev
  • 编译单个文件：tsc ./src/m1.ts
  • tsc编译整个项目：tsc

若要typescript编译整个项目，需要配置tsconfig.json文件。先使用tsc —init 自动生成配置文件，配置文件中的常用配置如下：

{
  "compilerOptions": {
    /* Visit https://aka.ms/tsconfig.json to read more about this file */

	// 编译的目标版本
    "target": "es5",   

    // 使用模块加载机制
	"module": "commonjs",
	  
	// 开启源代码映射，方便调试
	"sourceMap": true,    
    
	//编译结果目录
	"outDir": "./dist",      
    
	// 指明源代码所在目录
	"rootDir": "./src", 
    
    // 是否启用严格模式
	"strict": true, 
		// ...
	}
}

# 2. TypeScript变量作用域问题

如果不把文件当作模块使用的话TypeScript会认为所有文件里的代码都是在一个作用域里的，所以即使在不同文件也不能申明同名变量，解决办法：

  1. 把文件当作一个模块,在每个文件至少有一个export，若没有则在末尾加上：
export {}
  1. 使用namespace防止命名冲突，比如把每个文件里的代码包在一个namespace里
//a.ts
namespace A {
    let a = 1
}

//b.ts
namespace B {
    let a = 1
}

# 3. TypeScript类型注解

// 在非严格模式中，下面三种都允许为null
const a: string = 'foo'
const b: number = 100
const c: boolean = true

const f: null = null
const g: undefined = undefined
const e: void = undefined  // 一般用来标记函数没有返回值

注意编译target版本的问题：

const h: symbol = Symbol() // 如果目标文件是ES6之前会报错, 因为symbol是ES6才有

# 3.1 object类型

// 单独的object类型泛指除原始类型外的所有类型
const a: object = function(){} // {}、[]

# 3.2 数组类型

const arr1: Array<number> = [1,2,3]
const arr2: number[] = [1,2,3]

# 3.3 元组类型

元组类型是明确了元素数量及类型的数组，常用于函数返回多个返回值

const tuple:[number, string] = [1, 'foo']

# 3.4 枚举类型

JavaScript本身没有枚举类型，一般使用对象模拟

const enum logLevel {
	error,
	warn,
    info,
    debug
}

const enum method {
    'get' = 'GET',
    'post' = 'POST',
    'delete' = 'DELETE'
}

console.log(logLevel.error, logLevel.warn)           // 0 1
console.log(method.get, method.post, method.delete) // GET POST DELETE

# 3.5 函数类型

function f1(a: number, b: number=10, ...rest: number[]): string{
	return 'func1'
}

# 3.6 any类型

let a: any = 10 // ''、[]、{}...

/* any类型虽然不会报语法错误，但轻易不要使用，因为这就像没写类型约束一样可能产生运行时错误*/

# 4. 隐式类型推断

即使在typescript中没有使用类型注解，ts也会根据使用情况推断类型，如果无法判断则推断为any。

如果变量被推断为除any类型外的其他类型，则如果再被赋值为其他类型也会报错：

let age = 10 // 隐式推断为number类型
age = 'st'   // 报错

# 5. 类型断言

某些场景下需要明确告诉typescript变量的类型，类型断言不是类型转换。

类型断言的一个常见用例是当你从 JavaScript 迁移到 TypeScript 时：

const foo = {  };
foo.bar = 123; // Error: 'bar' 属性不存在于 ‘{  }’
foo.bas = 'hello'; // Error: 'bas' 属性不存在于 '{  }'

这里的代码发出了错误警告，因为 foo的类型推断为 {}，即没有属性的对象。因此，你不能在它的属性上添加 bar 或 bas，可以通过类型断言来避免此问题：

interface Foo {
  bar: number;
  bas: string;
}

const foo = {} as Foo;
foo.bar = 123;
foo.bas = 'hello';

在很多情景下，断言能让你更容易的从遗留项目中迁移，然而应该小心谨慎的使用断言，其被认为是有害的


# 6. 接口

抽象并约束对象的公共接口：

export interface Post {
    title: string
    content: string

    subtitle?: string          // **可选成员**
    readonly summary: string   // **只读成员**(初始化后只读)
    [key:string]: string       // **动态成员**(可动态添加任意成员)
    
}

const p: Post = {
    title:'t',
    content: 'hello',
    summary:'READONLY...'
}

抽象并约束类的公共接口：

export interface Eat {
    eat(food:string): void
}

export interface Run {
    run(distance:number): void
}

class Person implements Eat,Run {
	eat(food:string){
		console.log('person eat ', food)
	}

	run(distance:number)){
		console.log('person run ', distance)
	}
}

# 7. TypeScript中的类

TypeScript中除了可以使用Ecmascript中的类功能，还添加了一些额外的功能和用法，如类成员的访问修饰符、抽象类等。

# 7.1 属性在定义之前要先声明

可以在声明类型时直接赋值，不过一般在构造函数中赋值

class Post {
    title: string
    content: string

    constructor(title:string, content:string){
        this.title = title
        this.content = content
    }
}

const p = new Post('x', 'xcc')

# 7.2 访问修饰符

class Person {
    public name: string           // 共有属性（默认就是公有可省略，建议加上）
    private age: number           // 私有属性（只能内部访问，外部不可访问，子孙也不能用）
    protected gender: string      // 保护属性（只能内部和子孙内部使用）

	public readonly name: string  // 只读属性，初始化后不允许改变
		
}

# 7.3 抽象类

抽象类做为其它派生类的基类使用。 它们只能被继承，不会直接被实例化。 不同于接口，抽象类可以包含成员的实现细节。 抽象类内部可以通过abstract关键字定义抽象方法，继承的类必须实现抽象方法。

abstract class Animal {
	// 抽象方法
    abstract makeSound(): void;

	// 具体方法
    move(): void {
        console.log('roaming the earch...');
    }
}

抽象类中的抽象方法不包含具体实现并且必须在派生类中实现。 抽象方法的语法与接口方法相似。 两者都是定义方法签名但不包含方法体。 然而，抽象方法必须包含 abstract关键字并且可以包含访问修饰符。


# 8. 泛型

function creatNumberArrar(length: number, value: number): number[]{
    return Array(length).fill(value)
}
function creatStringArrar(length: number, value: string): string[]{
    return Array(length).fill(value)
}

console.log(creatNumberArrar(5,3)) // [ 3, 3, 3, 3, 3 ]
console.log(creatStringArrar(5,"f")) // [ 'f', 'f', 'f', 'f', 'f' ]

/*--------------------通过传入参数，函数经过泛型进一步抽象：------------------------*/

function creatArrar<T>(length: number, value: T): T[]{
    return Array<T>(length).fill(value)
}
console.log(creatArrar<number>(5,3)) 
console.log(creatArrar<string>(5,"f"))
