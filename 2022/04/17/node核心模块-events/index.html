Node核心模块—events

xucongcong
2022-04-17
  • Node

Node.js 的大部分核心 API 都是围绕惯用的异步事件驱动架构构建的，在该架构中，某些类型的对象（称为"触发器"）触发命名事件，使监听器函数被调用。

Node.js中很多内置模块继承EventEmitter类，例如：net.Server 对象在每次有连接时触发事件；fs.ReadStream 在打开文件时触发事件；stream在每当有数据可供读取时触发事件。

当 EventEmitter 对象触发事件时，所有绑定到该特定事件的函数都会被同步地调用。 被调用的监听器返回的任何值都将被忽略和丢弃。


# 1. EventEmitter类常用API

  • on：注册监听器，每次事件发生都会被触发，且多个监听器在事件触发后按注册顺序执行
  • once：也是注册监听器，但只会被触发一次
  • emit：触发事件
  • off：移除特定监听器
const Events = require('events')

const myEvent = new Events()

const callback1 = ()=> console.log('cb1 done')
const callback2 = ()=> console.log('cb2 done')

myEvent.on('task1', callback1)
myEvent.on('task1', callback2)

// 若使用once注册，则注册的callback只会被执行一次
// myEvent.once('task1', callback1)
// myEvent.once('task1', callback2)

// myEvent.off('task1', callback2) // 若移除，则callback2事件不会被执行
myEvent.emit('task1')
myEvent.emit('task1')

# 2. 将参数和this传递给监听器函数

  • 当调用普通的监听器函数时，标准的 this 关键字会被有意地设置为引用监听器绑定到的 EventEmitter 实例。
  • 可以使用 ES6 箭头函数作为监听器，但是，这样做时，this 关键字将不再引用 EventEmitter 实例。
const Events = require('events')

const myEvent = new Events()

myEvent.on('task', fucntion (...args) {
	console.log(args,this,this===myEvent)
})

myEvent.on('task', (...args)=> {
	console.log(args,this,this===myEvent)
})

myEvent.emit('task', 1, 2, 3, 4)

# 3. 同步执行vs异步执行

EventEmitter 按照注册的顺序同步地调用所有监听器。 这确保了事件的正确排序，并有助于避免竞争条件和逻辑错误。有的时候监听器函数可以使用setImmediate()或process.nextTick()  方法切换到异步的操作模式：

const myEmitter = new MyEmitter();
myEmitter.on('event', (a, b) => {
  setImmediate(() => {
    console.log('this happens asynchronously');
  });
});
myEmitter.emit('event', 'a', 'b');

# 4. error 事件

如果 EventEmitter 没有为 'error' 事件注册至少一个监听器，并且触发 'error' 事件，则会抛出错误，打印堆栈跟踪，然后退出 Node.js 进程。

const myEmitter = new MyEmitter();
myEmitter.emit('error', new Error('whoops!'));
// 抛出错误并使 Node.js 崩溃
1
2
3

作为最佳实践，应始终为 'error' 事件添加监听器。

const myEmitter = new MyEmitter();
myEmitter.on('error', (err) => {
  console.error('whoops! there was an error');
});
myEmitter.emit('error', new Error('whoops!'));
// 打印: whoops! there was an error

const { EventEmitter, errorMonitor } = require('events');

const myEmitter = new EventEmitter();
myEmitter.on(errorMonitor, (err) => {
  MyMonitoringTool.log(err);
});
myEmitter.emit('error', new Error('whoops!'));
// 仍然抛出错误并使 Node.js 崩溃

# 5. 捕获监听器函数的 Promise reject

监听器使用 async 函数是有问题的，因为它会在抛出异常的情况下导致未处理的promise：

const { EventEmitter } = require('events')
const ee = new EventEmitter();
ee.on('something', async (value) => {
  throw new Error('kaboom');
});
ee.emit('something')
1
2
3
4

6

EventEmitter 构造函数中的 captureRejections 选项或全局的设置可以在 Promise上安装 .then(undefined, handler) 句柄。 此句柄将异常异步地路由到 Symbol.for('nodejs.rejection') 方法（如果有的话）或 ['error'](http://nodejs.cn/api/events.html#error-events)事件句柄（如果没有）：

const { EventEmitter } = require('events')

const ee1 = new EventEmitter({ captureRejections: true });
ee1.on('something', async (value) => {
    throw new Error('kaboom');
});
ee1.on('error', console.log);

const ee2 = new EventEmitter({ captureRejections: true });
ee2.on('something', async (value) => {
    throw new Error('kaboom');
});
ee2[Symbol.for('nodejs.rejection')] = console.log;

setTimeout(() => {
    ee1.emit('something')
    setTimeout(() => {
        ee1.emit('something')
    }, 1000)
}, 1000)

# 6. 模拟实现发布订阅

class myPub{
    constructor(){
        this._events = {}
    }

    // 订阅事件
    subscribe(event, callback) {
        // 如果已有事件队列，则添加回调函数到该事件的执行队列
        if(this._events[event]){
            this._events[event].push(callback)
        // 如果没有此事件，则新建一个此事件的执行队列，并添加
        }else{
            this._events[event] = [callback]
        }
    }

    // 发布事件
    publish(event, ...args){
        // 只要队列存在且非空就执行，否则跳过
        if(this._events[event] && this._events[event]!==[])
            for(let callback of this._events[event])
                callback()
    }
}

const myEventPub = new myPub()

myEventPub.subscribe('task1',(...args)=>{ console.log('111=>',args) })
myEventPub.subscribe('task1',(...args)=>{ console.log('222=>',args) })

myEventPub.publish('task1', 1, 2, 3)
