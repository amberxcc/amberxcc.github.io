# 1. 模型设计基础

数据模型是一组由符号、文本组成的集合，用以准确表达信息，达到有效交流、沟通的目的。— 霍夫曼. 数据建模经典教程

# 1.1 数据模型设计的基本元素

  • 实体：描述业务的主要数据集合（如用户表、订单表）
  • 属性：描述实体内的单个信息（如用户表中的username字段）
  • 关系：描述实体与实体之间的数据规则：1-N，N-1，N-N

# 1.2 传统关系模型设计

概念模型 CDM 逻辑模型 LDM 物理模型 PDM
目的 描述业务系统要管理的对象 基于概念模型，详细累出所有实体、属性、关系 根据逻辑模型，结合数据库和物理结构，设计具体的表结构、字段、主外键
特点 用概念名词描述业务实体（如联系人） 基于业务的描述与数据库无关 与具体的数据库相关的实现细节
使用者 用户、需求分析师 需求分析师、开发者、架构师 开发者、DBA

# 2.1 文档模型设计的三个误区

  • 不需要文档模型设计？
  • MongoDB应该用一个超级大的文档来组织所有数据？
  • MongoDB不支持关联或事物？

# 2.2 MongoDB 无 Schema 的来源

严格来说，MongoDB同样需要概念、逻辑建模，只是文档模型设计的物理层结构可以和逻辑层类似，部分情况甚至可以直接使用逻辑层从而省略物理建模过程，这也是 MongoDB 无 schema 的说法来源。

# 2.3 文档模型设计的原则

文档模型没有第三范式的约束，两个主要的指导原则是性能和易用性。

# 3. 文档模型设计（三部曲）



# 3.1 基础设计

  1. 根据概念模型或业务需求推导出逻辑模型 - 找到对象
  2. 列出实体之间的关系 - 明确关系
  3. 套用逻辑设计原则来决定内嵌方式 - 进行建模
  4. 完成基础模型构建

示例：联系人管理

  • 1-1 关系建模：以内嵌为主，没有冗余（例外情况：大小超过16M）
  • 1-N 关系建模：以内嵌为主，用数组表示一对多，没有冗余，例外情况：嵌入后文档大小超过16M、数组长度太大、数组长度不确定。
  • N-N 关系建模：一般通过内嵌数组表示一对多、通过冗余实现N-N，不使用映射表。例外情况：嵌入后文档大小超过16M、数组长度太大、数组长度不确定。

# 3.2 工况细化

基于内嵌的关系模型可以根据实际情况换成引用方法：

  • 内嵌数组过长、长度不确定、频繁修改
  • 单个字段很大或不常用

💡 mongodb中的 $lookup 相当于关系型数据库中的LEFT JOIN，且没有数据库级别的限制，全靠业务逻辑维持数据一致性。且lookup的关联目标不能是分布式架构中的分片表。

# 3.3 参考设计模式

文档设计相当灵活，一个好的设计可以显著提升数据读写效率、降低存储资源需求。


例如IOT场景下对时序数据的分桶设计：假设正常情况下飞机每分钟产生一条飞行数据，格式：

{
	"id": "xxxxxxxxxxxx",
	"icao": "CA2790",
	"callsign": "CA2790",
	"ts": 1677663118774,
	"event": [
		"a": 12121,
		"b": 5534,
		"c": [110, 333]
			...
	]
}

由于每条数据的有用信息是event字段，其他信息都是重复信息，如果将一小时的数据放在一条记录中，就可以节省大量的重复信息存储和索引存储，同时查询效率更高：

{
	"id": "xxxxxxxxxxxx", //  重复字段
	"icao": "CA2790",     //  重复字段
	"callsign": "CA2790", //  重复字段
	"ts": 1677663118774,  //  重复字段
	"event": [
		[ "a": 12121, "b": 5534, "c": [110, 333] ...],
		[ "a": 12122, "b": 5535, "c": [111, 334] ...],
		...
	]
}

其他设计模式：

  • 记录文档版本：通过一个额外字段管理不同版本的文档
  • 近似计算：对于写入频繁但准确性要求不高的场景（如准确性要求不高的排名）
  • 预聚合：需要统计数量的字段使用单独维护，若每次都扫描动态查询，影响性能
