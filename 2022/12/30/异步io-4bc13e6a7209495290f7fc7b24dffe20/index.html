操作系统-异步IO模型与libuv

xucongcong
2022-12-30
  • 操作系统

I/O操作是计算机操作中速度最慢的一种，内存访问的时间是纳秒级，而磁盘和网络IO的访问时间则是毫秒级。不仅是速度，这两种操作的带宽差异也是这样，访问内存的带宽可以稳定在每秒几个GB，而磁盘和网络只能达到每秒几个MB，只有较好情况下才能达到每秒几个GB。

《UNIX 网络编程》 6.2 I/O Models 中介绍了五种网络 I/O 模型：

  • 阻塞 I/O
  • 非阻塞 I/O
  • I/O 的多路复用（select、poll、epoll）
  • 信号驱动的 I/O
  • 异步 I/O

# 1. 阻塞式I/O

在传统阻塞式I/O编程中，涉及I/O操作的函数会阻塞当前线程，只到I/O操作完成才能执行下一行代码，如下伪代码演示的就是传统的阻塞式socket操作：

// read操作会阻塞线程，只到数据可用为止
data = socket.read()

// 数据可用后才会执行后续代码
consume(data)

采用阻塞式I/O实现的Web服务器处理并发连接的方法是：每遇到一次请求，就新创建一条线程（或进程），这样的话，一个线程的阻塞操作就不会影响其他的连接了，如下图所示：

# 2. 非阻塞式IO

除了阻塞式IO，大多数操作系统还支持非阻塞时IO。在这种模式下，有关IO操作的函数不会阻塞进程，而是立即返回。处理这种IO模式，需要以轮训的方式主动查询资源是否准备好，伪代码如下：

resources = [socketA, socketB, fileA]

while(! resource.isEmpty()){
	for(resource of resources){
		data = resource.read() // 尝试读取
		if(data === NO_DATA_AVALIABLE){
			continue      // 没有资源准备好，进入下一次循环
		}else{
			consume(data) // 资源准备好，开始消费
		}
	}
}

相比阻塞式IO，使用非阻塞式IO的程序可以在一个线程里处理多个资源，但轮训算法依然浪费了大量的CPU时间。

# 3. IO多路复用

IO复用模型的思路是系统提供一种可以同时监控多个资源的操作，有了这个函数后，应用线程通过调用此函数就可以同时监控多个资源，被监控的资源中只要有一个准备就绪了，就会返回可读状态，这时询问线程再去通知处理数据的线程，对应线程此时再发起请求去读取数据，伪代码：

watchedList.add(socketA, FOR_READ)
watchedList.add(fileB, FOR_READ)

// 阻塞进程，每当有资源准备就绪时就返回事件
while (events = demultiplexer.watch(watchedList)) {
  for (event of events) {
    if (data === RESOURCE_CLOSED) {
      demultiplexer.unwatch(event.resource)
    } else {
      consumeData(data)
    }
  } 
}

// 在UNIX系统中的操作系统接口是select、poll、epoll，其中epoll是对前者的优化

IO多路复用的基本思路就是通过一个操作来监控多资源 ，达到不必为每个资源创建一个对应的监控线程，从而减少线程资源的开销。

# 4. 信号驱动的IO

I/O多路复用使用一个线程监控多个资源，减少了多线程监听的开销，但还是没有避免轮训算法的低效。所以就衍生了信号驱动IO模型。

信号驱动IO不是用轮训去监控资源就绪状态，而是在调用的时候建立一个信号联系，当内核数据准备好之后再通过信号通知线程，当线程收到信号后，此时再向内核发起读取数据的请求，因为信号驱动IO的模型下应用线程在发出信号监控后即可返回，不会阻塞，所以这样的方式下，一个应用线程也可以同时监控多个资源。

# 5. 异步IO

前面的几种IO方式，在数据从内核复制到应用缓冲的时候都是阻塞的，即应用先得知数据是否准备好然后再去读取数据。异步IO的优化思路是解决了应用程序需要先后发送询问请求、发送接收数据请求两个阶段的模式，在异步IO的模式下，只需要向内核发送一次请求就可以完成状态询问和数拷贝的所有操作。

# 6. 关于阻塞/非阻塞、同步/异步

阻塞、非阻塞：所谓阻塞就是发起读取数据请求的时，当数据还没准备就绪的时候，这时请求是即刻返回，还是在这里等待数据的就绪，如果需要等待的话就是阻塞，反之如果即刻返回就是非阻塞。

同步、异步：在IO模型里面如果请求方从发起请求到数据最后完成的这一段过程中都需要自己参与，那么这种我们就称为同步请求；反之，如果应用发送完指令后就不再参与过程了，只需要等待最终完成结果的通知，那么这就属于异步。

同步阻塞、同步非阻塞：他们不同的只是发起读取请求的时候一个请求阻塞，一个请求不阻塞，但是相同的是，他们都需要应用自己监控整个数据完成的过程。而为什么只有异步非阻塞 而没有异步阻塞呢，因为异步模型下请求指定发送完后就即刻返回了，没有任何后续流程了，所以它注定不会阻塞，所以也就只会有异步非阻塞模型了。

# 7. libuv

每个操作系统都有自己的多路复用接口，Linux的epoll，macOS的kqueue、Windows的IOCP。此外同一种I/O操作在不同类型的资源上，可能会有相当不同的行为，就算操作系统一样，还是会因为资源类型不同而产生差异。如UNIX操作系统中常规的文件不支持非阻塞操作，因此为了模拟非阻塞行为，必须使用相应的机制来模拟。

libuv是Node.js核心团队为了解决解决上述问题创建的异步IO原生库，除了把底层操作系统间的差异抽象掉，libuv还实现了reactor模式，因此它会提供一套API让开发者能够建立时间循环、管理事件队列、运行异步I/O操作，并将其他类型的任务加入队列
